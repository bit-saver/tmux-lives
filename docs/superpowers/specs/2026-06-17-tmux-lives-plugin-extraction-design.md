# tmux-lives plugin extraction — design (spec 1)

- **Date:** 2026-06-17
- **Status:** approved (open items resolved during review; GitHub account still TBD)
- **Scope:** Sub-project 1 of 2. Extract the existing tmux automation system from `~/.config/fish` into a standalone, public **fisher plugin** named **tmux-lives** at `~/workspace/tmux-lives`, with **no behavior change**. Sub-project 2 (macOS port) is a separate spec.

## Motivation

The tmux automation system (categorized session persistence, switcher, ghost-detach, restore, ShellFish coexistence) currently lives scattered across `~/.config/fish` (`conf.d/tmux.fish`, `custom/scripts/`, `~/.tmux.conf`, systemd units; documented in `docs/auto-tmux.md`). The user wants it as a self-contained, removable, version-controlled package — **tmux-lives** — that also installs on a MacBook. A fisher plugin gives clean install/update/remove and a real VCS home.

## Decisions (from brainstorming)

| Decision | Choice |
|---|---|
| Name | **tmux-lives** (repo + plugin + working dir `~/workspace/tmux-lives`) |
| Path | Extract on Linux first (parity-gated), then port to macOS (separate spec) |
| Form | Public GitHub repo, installed as a fisher plugin |
| Install UX | `fisher install <owner>/tmux-lives && tmux-setup` |
| Packaging | fisher deploys the fish bits; bundled `tmux-setup`/`tmux-teardown` handle the non-fisher glue; `tmux-status` reports the whole system |
| Genericness | No host-specifics in the plugin (paths, uid, host names, reboot-cadence assumptions are parameterized or detected) |
| Behavior | **Unchanged** in spec 1 — pure repackaging; all four test suites stay `ALL PASS` |

## Why fisher + a setup/teardown pair (not a hand-rolled installer)

fisher only deploys `.fish` files under `conf.d/ functions/ completions/ themes/`, fetched as a tarball over curl (no `git clone`), and offers no post-install hook. So:

- The fish bits ride fisher (which also gives precise, tested removal — the riskiest part of a hand-rolled uninstaller, eliminated).
- The non-fisher glue (service unit, `~/.tmux.conf` fragment, TPM) is handled by bundled functions `tmux-setup`/`tmux-teardown`.
- Visibility comes from `tmux-status`, which reports the *whole* system — `fisher list` only sees the fish half.

## Architecture

### Repo layout

```
tmux-lives/
├── conf.d/
│   └── tmux.fish              # trigger/gate, pick, prune, restore+dispose, wrappers, ts, hooks (as today)
├── functions/
│   ├── tmux-categorize.fish   # the categorizer (moved scripts/→functions/ so fisher deploys it)
│   ├── tmux-setup.fish        # generate+install service unit; place+source tmux.conf fragment; ensure TPM plugins
│   ├── tmux-teardown.fish     # reverse tmux-setup
│   └── tmux-status.fish       # full-system doctor (fish bits, categorizer, service state, fragment, TPM, resurrect)
├── tmux/
│   └── tmux-lives.conf.in       # ~/.tmux.conf fragment template (absolute paths filled in by tmux-setup)
├── tests/
│   └── test-*.fish             # the four suites — run from the repo, not deployed
├── README.md
├── CLAUDE.md                   # plugin-dev guide (already stubbed in the repo)
└── docs/superpowers/specs/…    # this design + lineage
```

### Component responsibilities

- **`conf.d/tmux.fish`** — unchanged logic; the only edits are to reference the categorizer at its deployed path (see below) instead of `custom/scripts/`.
- **`functions/tmux-categorize.fish`** — same categorize/rename/menu logic, relocated. Still invoked as a script under `fish --no-config` from tmux `#()`; it just lives at a fisher-deployed path now.
- **`tmux-setup`** — idempotent. Resolves `$__fish_config_dir`, renders `tmux/tmux-lives.conf.in` → `~/.config/tmux/tmux-lives.conf` with absolute paths baked in, adds a single `source-file` line to `~/.tmux.conf` (guarded against duplicates), generates+installs the user service unit (systemd on Linux — launchd is spec 2), and ensures TPM + required tmux plugins.
- **`tmux-teardown`** — removes the service unit, the `source-file` line, and the generated fragment. (Then `fisher remove` deletes the fish files.)
- **`tmux-status`** — prints each layer's state so install/health is visible end to end.

### The path-portability detail

tmux's `status-right` needs an *absolute* path to the categorizer, but fisher deploys to `$__fish_config_dir` (which varies / is XDG-dependent). `tmux-setup` runs in fish, knows that dir, and bakes the resolved path into the generated `tmux-lives.conf`. So the fragment is always correct regardless of where fisher installed.

## Scope boundary — what's IN vs OUT

**IN** (the tmux-lives system):
- `conf.d/tmux.fish`, `tmux-categorize.fish`, the four logic areas, the ghost-detach/restore/commandeer behavior.
- The **tmux-side** ShellFish coexistence: the `commandeer` hook (already in `tmux.fish`) and the `LC_TERMINAL`/`update-environment` passthrough (goes in the fragment).
- The systemd save/boot-restore units (generated by `tmux-setup`).

**OUT** (stays in the user's personal `~/.config/fish`, not the plugin):
- `conf.d/shellfish.fish` clipboard/sharesheet helpers (`pbcopy`/`pbpaste`/…) — a separate concern, used by `custom.fish`, not tmux-specific.
- Anything host-specific (see Genericness).

**Resolved during review:** `test-shellfish.fish` currently tests *both* the clipboard helpers (OUT) and the tmux `LC_TERMINAL` passthrough (IN). The `LC_TERMINAL`-passthrough assertions **move into the plugin's test suite**; the clipboard-helper assertions stay with the user's `~/.config/fish`.

## Genericness requirement (de-hardcode audit)

Spec-1 extraction must remove/parameterize host-specifics so the public plugin is portable. Known items to audit:
- Absolute user paths and `bitsaver`/home assumptions.
- `install-tmux-save-unit.sh`'s `After=user@1000.service` (hardcoded uid) → derive uid at setup time.
- Proxmox "nightly reboot" assumptions baked into defaults/comments → keep behavior, neutralize the host-specific rationale.
- Any `LANG=C`/host-locale workarounds → keep as defensive defaults, not host assertions.

## Tests / parity gate

The four suites (`test-tmux-auto`, `test-tmux-restore`, `test-tmux-categorize`, `test-shellfish`) move to `tests/`, adjusted only to source the plugin's `conf.d`/`functions` from the repo (they already isolate via `-L` sockets and suppress the trigger). **Success criterion:** after `fisher install <repo>` + `tmux-setup`, behavior is identical to today and all four suites print `ALL PASS`. No behavior change is in scope for spec 1.

## Claude context migration (cutover)

When the user switches their primary working dir to `~/workspace/tmux-lives`, replicate the Claude context with the migration skill (command already staged in `~/workspace/tmux-lives/CLAUDE.md`):

```bash
rm -rf ~/.claude/projects/-home-bitsaver-workspace-tmux-lives
bash ~/.claude/skills/migrating-project-directories/scripts/claude-code-migrate.sh \
     --mode replicate ~/.config/fish ~/workspace/tmux-lives
```

Run last, so it captures the post-extraction state. `~/.config/fish` is left intact; claude-mem history stays under the `fish` label and is reachable via the CLAUDE.md search nudge.

## Cross-platform seam (implemented in spec 2)

Only `tmux-setup`/`tmux-teardown`/`tmux-status` branch on OS (systemd ↔ launchd, paths, service enable/disable). Everything else — `conf.d/tmux.fish`, the categorizer, the fragment — is OS-agnostic. macOS also drops the nightly-reboot premise (laptops sleep), which shapes whether boot-restore is even wanted there. Both are spec-2 concerns.

## Success criteria (spec 1)

1. `fisher install <local clone or repo>` + `tmux-setup` reproduces today's behavior exactly.
2. All four test suites print `ALL PASS` from the repo.
3. `tmux-teardown` + `fisher remove` cleanly removes everything (verified by `tmux-status` reporting a clean state).
4. No host-specifics remain in the plugin (genericness audit complete).
5. `~/.config/fish` continues to work unchanged until cutover.
